# M-0 Implementation Plan: Repository Bootstrap

> **Milestone Objective:** To lay the foundational plumbing for the `Arth` project.

By the end of this milestone, we will have a fully structured, version-controlled project skeleton. This isn't just about creating empty folders; it's about setting up the professional-grade tooling that will ensure our code is clean, our dependencies are managed, and our application is buildable and testable from day one. It’s the key to moving fast without breaking things.

---

### Success Criteria

1.  **Repo Ready:** A local `git` repository is initialized and pushed to a new private GitHub repository with the `main` branch protected.
2.  **Structure Built:** The project's directory structure is created according to the TRD.
3.  **Dependencies Managed:** A `pyproject.toml` file is created to manage all our Python dependencies and tool configurations.
4.  **Containerized:** `Dockerfile` and `docker-compose.yml` files are created, allowing the entire application and its database to be built and run with a single command.
5.  **CI Pipeline Green:** A GitHub Actions CI pipeline is configured to automatically check code quality (lint, format, type-check) on every push, ensuring our main branch stays healthy.
6.  **DB Ready to Evolve:** The database migration tool, `Alembic`, is initialized, ready for us to define our first database schema.

---

### Step-by-Step Implementation Plan

#### Step 1: Initialize Git & GitHub Repository

*   **✅ Status: Completed**
*   **What:** We'll start by creating a local Git repository and a corresponding private repository on GitHub. Git is our time machine, allowing us to track every change and roll back if we need to. GitHub is the cloud home for our code.
*   **How:**
    1.  Initialize the local repository.
    2.  Create a new **private** repository named `arth` on your GitHub account.
    3.  Link the local repository to the remote one on GitHub.
    4.  Create a `.gitignore` file. This is a list of files that Git should ignore (like temporary files, caches, and secrets). We'll start with a standard Python template.
    5.  Make our first commit and push it to GitHub.
*   **Post-setup Task:** On the GitHub repository settings, we'll set up a **branch protection rule** for `main`. This will prevent anyone (including us!) from pushing broken code directly to our main branch. It's a simple safety net that enforces good habits.
    *   **Update:** GitHub's free plan for private repositories restricts this feature. We are adopting an "Honor System" Pull Request workflow instead, which achieves the same process goal.

#### Step 2: Create Project Directory Structure

*   **What:** We'll create the folders and empty files for our project, exactly as laid out in the TRD's directory structure diagram.
*   **Why:** A clean, logical structure makes a project a joy to work on. It ensures we know exactly where to put new code and where to find existing code, which is essential for maintainability.

#### Step 3: Set Up Python Project with `pyproject.toml`

*   **What:** We'll create a `pyproject.toml` file at the root of our project.
*   **Why:** This is the modern centerpiece for managing a Python project. It will define our project's name, version, and, most importantly, all the libraries (dependencies) it needs to run. We'll also use it to configure our development tools like `Ruff`, `Black`, and `mypy`, keeping all our configuration in one predictable place.

#### Step 4: Bootstrap the Database with Alembic

*   **What:** We will initialize `Alembic`.
*   **Why:** Our database schema (the tables, columns, and relationships) will change as we build new features. Alembic is a tool that lets us manage these changes in a structured, version-controlled way. Think of it as Git for your database schema. It's an indispensable tool for any database-driven application. We'll configure it to work with our `SQLModel` definitions.

#### Step 5: Containerize the Application with Docker

*   **What:** We'll write two crucial files: `Dockerfile` and `docker-compose.yml`.
*   **Why:**
    *   The `Dockerfile` is a recipe for creating a "container image" of our application. This image is a self-contained package that includes our code and all its dependencies. It guarantees that our app runs the same way on your laptop as it will in the cloud.
    *   `docker-compose.yml` is an orchestration script. It will allow us to define and run our entire application stack—the Python app (`app`) and the Postgres database (`db`)—with a single, simple command: `docker-compose up`. This makes setting up a development environment incredibly easy and reproducible.

#### Step 6: Configure the CI Pipeline with GitHub Actions

*   **What:** We'll create a workflow file under `.github/workflows/ci.yml`.
*   **Why:** This is our automated quality gatekeeper. Every time we push code to GitHub, this Continuous Integration (CI) pipeline will automatically:
    1.  Check out the code.
    2.  Install all dependencies.
    3.  Run `Ruff` and `Black` to ensure the code is well-formatted and free of common errors (linting).
    4.  Run `mypy` to check for type errors, which helps prevent a whole class of bugs.
    5.  Run our test suite with `pytest` (it will be empty at first, but the step will be there).
*   If any of these steps fail, GitHub will alert us immediately. This feedback loop is the secret to maintaining a high-quality, stable codebase over time.
